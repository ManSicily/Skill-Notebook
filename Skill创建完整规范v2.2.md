> **面向读者**：Skill 创作者（AI）、Skill 使用者（人）、团队协作者
> **目标**：更快、更省 token、更少返工地完成 Skill 开发
> **来源**：v2.0 复盘 + 第一性原理 + 用户实测反馈
> **核心变化（vs v2.1）**：示例中的业务术语全部替换为通用占位符，可直接对外分享

## 快速入门（必读）

> 如果你只有 5 分钟，读这一章就够了。

### 七 Phase 一句话总结

| Phase | 名字 | 做什么 | 谁主导 | 可以跳过？ |
|-------|------|--------|--------|-----------|
| 0 | 意图对齐 | 说清楚要做什么 | 人 | ❌ 不可跳过 |
| 1 | 规范定义 | 写 SKILL.md（输入/输出/规则） | AI + 人 | ❌ 不可跳过 |
| 2 | 示例数据验证 | 用手工方式验证规范对不对 | 人 + AI | A 类可跳过 |
| 3 | 脚本实现 | 按规范写代码 | AI | ❌ 不可跳过 |
| 4 | 示例数据测试 | 用示例数据跑脚本 | AI + 人 | A 类可跳过 |
| 5 | 真实数据验证 | 用真实数据跑完整流程 | AI + 人 | ❌ 不可跳过 |
| 6 | 交付与存档 | 打包、版本号、知识沉淀 | AI | ❌ 不可跳过 |

### A/B/C 类一句话判断

| 类型 | 触发特征 | 规范重量 | 可跳过的 Phase |
|------|---------|----------|---------------|
| 🟢 A 类 | 一次性的简单操作 | 轻量 | Phase 2、Phase 4 |
| 🟡 B 类 | 周期性执行、多步处理 | 标准 | 无 |
| 🔴 C 类 | 多源数据、多文件联动 | 完整 | 无 |

### 两条铁律

1. **Phase 0-2 没完成，不进 Phase 3**（规范没确认就写代码 = 浪费时间）
2. **Phase 3 没完成，不进 Phase 4**（脚本都没写，测什么？）

## 决策树（纯文字版）

### 决策树 1：A/B/C 分类（5 个问题）

```
Q1：这个 Skill 会执行几次？
  ├─ 只执行一次 → 跳到 Q5
  └─ 会执行多次 → Q2

Q2：有几步处理逻辑？
  ├─ 1 步 → A 类（轻量）
  ├─ 2-4 步 → Q3
  └─ 5 步以上 → C 类（重量）

Q3：有几个交付物？
  ├─ 1 个 → B 类（中量，偏轻）
  ├─ 2-4 个 → Q4
  └─ 5 个以上 → C 类（重量）

Q4：需要接入外部系统吗？（数据平台后台、企业微信、API 等）
  ├─ 不需要 → B 类（中量）
  └─ 需要 → C 类（重量）

Q5：复杂度兜底检查
  ├─ 逻辑简单、输出单一 → A 类
  └─ 有一定复杂度 → B 类
```

### 决策树 2：哪些 Phase 可以跳过？

```
当前 Skill 类型是？
  ├─ A 类 → 可跳过 Phase 2（示例数据验证）和 Phase 4（示例数据测试）
  │         原因：逻辑简单，规范不容易写错
  ├─ B 类 → 不可跳过任何 Phase
  └─ C 类 → 不可跳过任何 Phase，且 Phase 2 和 Phase 4 必须认真做
```

```

结果：
```

## 一、为什么需要这本规范

### 经验数据（来自真实 Skill 复盘）

| 指标 | 数据 |
|------|------|
| 看板版本迭代次数 | 66 次（v1.0 → v1.68） |
| 数据校验失败次数 | 3 次（每次都因不同原因） |
| 核心 bug 数量 | 9+ 个（逻辑 bug，非语法 bug） |
| 交付物清单变更次数 | 5 次 |
| **根因** | **没有在动手之前把规范定义清楚** |

### 规范的目标

```
目标 1：减少返工 — 规范定义清楚后再动手
目标 2：省 token — 一次说清楚，不来回补
目标 3：少犯低级错误 — 校验和自验内嵌到流程里
目标 4：可维护 — 规范版本化，变更可追溯
目标 5：复杂度匹配 — 不同复杂度的 Skill 用不同重量的规范
```

## 二、Skill 创作的生命周期（七 Phase）

```
Phase 0：意图对齐       ← 开始动手前必须完成
Phase 1：规范定义       ← 最重要的阶段（按复杂度分层）
Phase 2：示例数据验证   ← 手工验证规范（不写代码）
Phase 3：脚本实现       ← 有了规范再写代码
Phase 4：示例数据测试   ← 用示例数据测试脚本
Phase 5：真实数据验证   ← 用真实输入做完整验证
Phase 6：交付与存档     ← 打包、版本化、知识沉淀
```

**两条铁律**：
1. Phase 0-2 没完成，不要进入 Phase 3。提前动手 = 浪费时间。
2. Phase 3 没完成，不要进入 Phase 4。脚本都没写，测什么？

> 💡 **为什么从六 Phase 改成七 Phase？**
> 原 Phase 2（示例数据验证）存在逻辑矛盾：还没写脚本，怎么"跑数据验证"？
> 解决办法：拆成两个独立 Phase——
> - Phase 2：手工验证规范（不写代码）
> - Phase 4：用示例数据测试脚本（脚本写完后）
> 逻辑通顺，每个 Phase 职责单一。

## 三、Skill 复杂度分层

> **核心原则**：Skill 规范是给不同复杂度的任务准备的，不是给所有任务硬套的同一套模板。
> 复杂度越高 → 规范越重，自验越严，目录结构越清晰。

### 🟢 A 类 · 轻量 Skill

**触发特征：** 单次操作、触发频率低、逻辑简单、输出单一  
**典型场景：** 按条件旋转 PDF、筛选 Excel 某几行、发送格式化的邮件  
**规范要求：**
- 无需写独立 Spec，直接在 SKILL.md 里描述处理逻辑
- 脚本数量：1 个
- 交付物：1 个（文件或消息）
- 自验：脚本末尾输出关键数字，人工确认即可
- 版本管理：简单记录变更内容即可
- **Phase 2（示例数据验证）可跳过**，Phase 4（示例数据测试）也可跳过

---

### 🟡 B 类 · 中量 Skill

**触发特征：** 每月/每周固定执行、多步处理、输入输出有固定格式  
**典型场景：** 月度数据汇总、周期性报告生成、榜单抓取、发票整理  
**规范要求：**
- 需要写 Spec（B 类标准三段式，见第五章）
- 脚本数量：2–4 个（按步骤或模块拆分）
- 交付物：2–4 个
- 自验：脚本末尾输出结构化自验清单，包含输入完整性和业务逻辑校验
- 版本管理：SKILL.md 和交付物分开版本化
- **Phase 2 和 Phase 4 都不可跳过**

---

### 🔴 C 类 · 重量 Skill

**触发特征：** 多源数据整合、多文件联动、需要业务理解、频繁变更风险高  
**典型场景：** 月度数据复盘、多源数据对账、跨系统活动运营自动化  
**规范要求：**
- 必须写完整 Spec，经过用户确认再开发
- 脚本数量：5 个以上（按处理步骤纵向拆分）
- 交付物：5 个以上
- 自验：自动对比上月数据 + 异常阈值报警 + 完整数字清单
- 版本管理：严格版本化，变更记录到 SKILL.md 头部
- **必须建立数据和代码分离的目录结构**
- **Phase 2、Phase 4、Phase 5 都不可跳过**

---

### 复杂度判断检查清单

在进入 Phase 1 之前，先判断 Skill 类型：

```
[ ] 这个 Skill 会做几次？（一次 → A 类；多次 → B/C 类）
[ ] 这个 Skill 有几步处理逻辑？（1 步 → A 类；2-4 步 → B 类；5 步以上 → C 类）
[ ] 这个 Skill 有几个交付物？（1 个 → A 类；2-4 个 → B 类；5 个以上 → C 类）
[ ] 这个 Skill 需要接入外部系统吗？（不需要 → A/B 类；需要 → C 类）
[ ] 这个 Skill 的数据逻辑复杂吗？（不复杂 → A/B 类；复杂，需要业务判断 → C 类）
```

**判断结果**：
- 大部分是"一次、1 步、1 个交付物" → **A 类**
- 大部分是"多次、2-4 步、2-4 个交付物" → **B 类**
- 大部分是"多次、5 步以上、5 个以上交付物、需要接入外部系统" → **C 类**

### 4.2 AI 做什么

收到 Phase 0 需求后，AI **只做一件事**：

> 用大白话复述一遍你的需求，让你确认理解得对不对。

**模板**：

```
我理解的是这样的：
- 名称：___ 
- 目标：___
- 触发词：___
- 结束条件：___
- 和已有 ___ Skill 的区别：___
- 频率：___
- 复杂度预估：___（A 类 / B 类 / C 类）

请确认我的理解是否正确。
如果不对，请告诉我哪里不对，我重新理解。
如果对，我进入 Phase 1：规范定义。
```

### 4.3 意图对齐检查清单

在进入 Phase 1 之前，必须同时满足：

- [ ] AI 复述了用户需求
- [ ] 用户确认"对"（不是"差不多"）
- [ ] 没有悬而未决的问题
- [ ] AI 和用户对"做好了"的标准达成一致
- [ ] AI 给出了复杂度预估（A/B/C 类）

> ⚠️ **警告**：如果用户说"先做着吧，做完再说"，AI 应该拒绝并说：
> "为了减少返工，建议我们先把规范对齐清楚再做。你能花 5 分钟确认一下我理解得对不对吗？"

### 5.3 SKILL.md 第二章模板（输入文件规范）

#### A 类模板（简单描述）

```markdown

- 文件：xxx.xlsx，来自 [来源]

2. 输出字段：[保留哪些字段]
```

#### B 类模板（标准三段式）

```markdown
## 二、输入文件规范

### 必填文件（用户提供）
| 文件名 | 来源 | 字段要求 | 异常处理 |
|--------|------|----------|----------|
| xxx.xlsx | [来源] | A/B/C 三列必填 | 缺失 → 报错 |

### 选填文件
| 文件名 | 说明 |
|--------|------|
| yyy.xlsx | 如果有则传，没有则跳过 |

### 文件名校准规则
- 年月必须在文件名中（如 "2026-03"），允许顺序颠倒
- glob 匹配模式：`*{year}*{month:02d}*`（不要求关键字在特定位置）
- 如果文件名不符合规范 → 脚本停止，提示用户重命名

### 数据校验规则
1. **完整性**：必填文件是否都存在
2. **数字一致性**：[具体数字关系，如"三张表的积分总数必须完全一致"]
3. **逻辑关系**：[具体逻辑关系，如"数据明细行数 ≈ 奖励明细行数（允许差异 ≤ 5）"]
4. **异常报警**：[具体阈值，如"数据总量与上月差异 > 30% → 提醒用户确认"]
```

## 六、Phase 2：示例数据验证（手工验证，不写代码）

### 6.1 为什么需要 Phase 2

Skill 规范（SKILL.md）写完了，但可能写得不对、写得不完整。  
Phase 2 的目标是在**不写代码**的情况下，发现规范的问题。

**示例数据会暴露规范的问题**：
- 文件列名和规范不一致
- 数据口径差异（差 3-5 条是正常的，但规范没写）
- 特殊情况的处理方式（比如"记录名称为空"怎么办？）

### 6.2 用户做什么

从真实数据中拿出 ≤ 10 条，**脱敏**后发给 AI：

- 数字改掉（金额、具体人名）
- 名称改成通用描述（如"某条目A"）
- 保留数据结构和逻辑关系不变

> ⚠️ **注意**：Phase 2 是**手工验证规范**，不写代码，不跑脚本。

### 6.3 AI 做什么（不写代码，只验证逻辑）

**AI 收到示例数据后，只做逻辑验证，不写脚本**：

1. **检查文件列名**：示例数据的列名，跟规范里写的"必填列"对得上吗？
2. **检查边界情况**：示例数据有边界情况（如空值、重复值），规范里写清楚怎么处理了吗？
3. **检查输出格式**：示例数据的输出，跟规范里写的"输出格式"对得上吗？
4. **检查业务逻辑**：规范里的业务逻辑（如"积分 = 基础分 × 2"）跟示例数据一致吗？

**如果有问题** → 回到 Phase 1，修正 SKILL.md  
**如果没问题** → 进入 Phase 3（脚本实现）

### 6.4 Phase 2 完成标准

```
[ ] 示例数据已提供（≤10 条，脱敏）
[ ] AI 验证了规范的逻辑完整性（没有写代码、没有跑脚本）
[ ] 发现的问题已全部修正（回到 Phase 1）
[ ] 用户说了"规范没问题，可以写代码了"
→ 进入 Phase 3
```

```python

prev = load_previous_month_data()   # 自动加载上月数据
curr = calculate_current_month()    # 本月数据

# 计算变化（区分绝对值和百分比）
abs_changes = {
    "记录总量": curr.total - prev.total,
    "积分总量": curr.points - prev.points,
}
pct_changes = {
    "记录总量变化率": (curr.total - prev.total) / prev.total * 100 if prev.total else 0,
    "积分总量变化率": (curr.points - prev.points) / prev.points * 100 if prev.points else 0,
}

print("--- 绝对值变化 ---")
for key, value in abs_changes.items():
    flag = "🔴" if abs(value) > 100 else "⚠️" if abs(value) > 50 else "✅"
    print(f"{flag} {key}：{value:+d}")

print("--- 环比变化率 ---")
for key, value in pct_changes.items():
    flag = "🔴" if abs(value) > 30 else "⚠️" if abs(value) > 20 else "✅"
    print(f"{flag} {key}：{value:+.1f}%")

# 异常阈值报警
if abs(pct_changes["记录总量变化率"]) > 30:
    print("⚠️ 记录总量环比变化超过 30%，请确认是否正常")
```

### 示例 1：输入文件规范

#### ❌ 坏规范（模糊、不可验证）

#### ✅ 好规范（具体、可验证）

```markdown
## 二、输入文件规范

### 必填文件
| 文件名 | 来源 | 必填字段 | 异常处理 |
|--------|------|----------|----------|
| 数据明细表.xlsx | 用户上传 | 记录ID / 用户ID / 等级 / 积分 | 缺失 → 报错停止 |
| 奖励明细表.xlsx | 用户上传 | 记录ID / 奖励金额 / 状态 | 缺失 → 报错停止 |

### 文件名校准规则
- 文件名必须包含年月（如 "2026-03"），允许顺序颠倒
- glob 匹配：`*{year}*{month:02d}*`
- 不符合规范 → 停止，提示用户重命名

### 数据校验规则
1. **完整性**：两个文件都必须存在，否则停止
2. **数字一致性**：数据明细表的积分总和 = 奖励明细表的积分总和
3. **逻辑关系**：数据明细表的行数 ≥ 奖励明细表的行数（一条记录可能对应多笔奖励）
4. **异常报警**：记录总量与上月差异 > 30% → 提醒用户确认

### 合并规则
- 按 `记录ID` 字段 left join（保留数据明细表的所有行）
- 重复记录ID（同一记录多笔奖励）→ 奖励金额 sum 聚合
- 匹配不到奖励的记录 → 积分填 0，标注"未处理"
```

## 十三、用户下指令的正确方式

### 13.1 分层下指令

| 层级 | 说明 | 用户怎么说 | AI 怎么接 |
|------|------|-----------|-----------|
| **方向层** | 我要做什么 | "帮我做个数据分析 Skill" | AI 复述 → 确认意图 |
| **规范层** | 输入/输出/规则是什么 | "输入是 5 个文件，输出是 5 个交付物" | AI 写入 SKILL.md |
| **执行层** | 具体改什么 | "把等级列改到名称前面" | AI 改代码 |
| **校验层** | 怎么验收 | "跑完后核对记录总量和系统数据" | AI 输出自验清单 |

**规则**：层级从高到低，同一次对话不要跨越超过 2 个层级。

### 13.2 反馈的正确方式

每次 AI 做完，用户不要只说"好"或"不对"。按这个格式反馈：

```
[对/不对]：具体是哪个文件的哪个数字/格式/内容有问题 + 正确值/正确格式是什么
```

**例子**：

- ✅ "对，记录总量对了"
- ✅ "不对，邮件正文里的记录总量应该填 155，不是 143"
- ✅ "不对，看板的高等级数量应该是 5，不是 4"
- ❌ "不对"（没有说清楚哪里不对）

## 十四、异常情况处理

| 情况 | AI 怎么处理 |
|------|------------|
| 用户说"先做着吧，做完再说" | 拒绝："为了减少返工，建议先确认规范。你能花 5 分钟看看我写的 SKILL.md 吗？" |
| 用户一次提了很多改动 | 拆解：先做 A，完成后再做 B，不要一次做一堆 |
| 脚本跑出来数字和预期差很多 | 先查代码逻辑，不要直接让用户重新上传文件 |
| 发现和已有 Skill 撞了 | 先停下来问用户："这个和 \_\_\_ Skill 是什么关系？合并还是分开？" |
| 用户给的数据有误 | 指出哪个文件有问题，而不是默默用错误数据继续跑 |
| Phase 2 验证失败 | 回到 Phase 1，修正 SKILL.md，不要直接改脚本 |
| Phase 4 测试失败 | 回到 Phase 3，修正脚本，然后重新跑 Phase 4 |
| Phase 5 验证失败 | 回到 Phase 3，修正脚本，然后重新跑 Phase 4 → Phase 5 |

## 十五、快速参考（整合版）

### 15.1 AI 启动检查清单（每次写 Skill 前先读这个）

```
□ 判断 Skill 类型：A / B / C（用决策树 1）
□ 确认是否需要写 Spec（A 类跳过，B/C 类必须写）
□ 确认 Phase 0 意图对齐完成（用户说了"对"）
□ SKILL.md 必写章节写完了吗？（根据复杂度选择，见第五章）
□ 用户说了"对"吗？（不是"差不多"）
□ 有示例数据验证过吗？（如果是新 Skill，且是 B/C 类）
□ 真实数据验证了吗？（如果是更新已有 Skill）
□ 自验清单输出过了吗？
□ 打包里有没有真实数据？
□ 版本号更新了吗？
```

### 15.2 用户给 AI 反馈的快捷模板

```
对/不对：[文件/脚本名称]
哪里：[具体内容]
应该是：[正确值/正确格式]
```

### 15.3 新建 Skill 检查清单

```markdown
## 新建 Skill 前检查清单

- [ ] 判断 Skill 类型：A / B / C（用决策树 1）
- [ ] 确认是否需要写 Spec（A 类跳过，B/C 类必须写）
- [ ] 确认脚本数量（参考第五章的复杂度分层）
- [ ] 确认交付物清单（开发前写死，开发中不增加）
- [ ] 确认自验方式（A/B/C 类对应的自验机制）
- [ ] 确认目录结构（C 类必须数据和代码分离）
- [ ] SKILL.md 写完后读给用户听，用户确认"对"再进入代码环节
- [ ] 代码写完后用示例数据自验（Phase 4），输出自验清单再发给用户
- [ ] 交付物版本号记录到 SKILL.md 头部
```

### 15.4 Phase 完成标准总表

| Phase | 完成标准（简化） |
|-------|----------------|
| Phase 0 | AI 复述需求 + 用户说"对" + 复杂度预估 |
| Phase 1 | SKILL.md 必写章节写完 + 用户说"对" |
| Phase 2 | 示例数据验证通过 + 用户说"可以写代码了"（A 类可跳过） |
| Phase 3 | 所有脚本写好 + 每个脚本都有自验清单框架 |
| Phase 4 | 示例数据跑通 + 自验清单有真实数字 + 用户说"对"（A 类可跳过） |
| Phase 5 | 真实数据验证通过 + 用户说"对" |
| Phase 6 | 打包完成 + 版本号更新 + 存档完成 + 无真实数据 |

## 十六、核心原则总结

> **Skill 规范是给不同复杂度的任务准备的，不是给所有任务硬套的同一套模板。**
> 
> 复杂度越高 → 规范越重、自验越严、目录结构越清晰。
> 复杂度越低 → 简化流程，直接在 SKILL.md 里写逻辑即可。
> 
> **最底线原则：** 把 AI 当成一个每次都会犯错的执行者，而不是理解意图就不会错的智者。越是这样假设，规范写得越细，校验做得越全，返工越少。
> 
> **另一条底线原则：** Phase 0-2 没完成，不要进入 Phase 3。提前动手 = 浪费时间。
> 
> **第三条底线原则：** Phase 3 没完成，不要进入 Phase 4。脚本都没写，测什么？
